# GitHubとXRPLを連携したOSSトークン化PF仕様書

## 1. プロジェクト概要

本プロジェクトは、OSSプロジェクト向けにトークンを活用した資金調達・支援者報酬・ユーティリティ提供を行うWeb3プラットフォームを提供する。OSSリポジトリごとにトークンを発行し、そのトークンを通じて支援・参加・インセンティブ循環を行い、OSSの持続的な発展を支える。

## 2. 対象ユーザー

- OSSリポジトリのメンテナー・責任者
- OSS貢献者（コントリビュータ）
- OSS利用者・支援者（開発者、企業など）

## 3. トークン設計

### 3.1 トークン生成モデル（メンテナー初回発行なし）

- メンテナーは**初回にトークンを保有しない**。
- プロジェクト登録時にトークン名・単位などの**トークン初期設定**のみを行う。
- **最大供給量は設定しない**（無制限発行）。寄付に応じて随時トークンを発行するため、供給量上限による制約を回避。
- トークンの発行は**支援金の受け取りや貢献活動**に応じて、必要な分だけ新規発行される。
- 将来的にインフレ抑制が必要な場合は、**発行レート調整**や**バーン機能の強化**で対応。

### 3.2 プレマネー評価方式（自然な価格上昇モデル）

#### 3.2.1 価格算出の基本方針

- トークン価格は**寄付総額**と**GitHubリポジトリ情報**を組み合わせて自動算出
- プロジェクトの成長（寄付増加、GitHub活動活発化）に伴い価格が自然に上昇
- 早期支援者ほど安価でトークンを取得でき、**アーリーアダプターメリット**を享受

#### 3.2.2 具体的な価格算出式 例

```text
トークン価格 = 基準価格 × (1 + 寄付総額係数) × (1 + GitHub活動係数)

寄付総額係数 = log(累積寄付額 / 10万円) × 0.1
GitHub活動係数 = (スター数/1000 + コミット数/100 + Issue数/50) × 0.05
```

#### 3.2.3 価格上昇の合理的根拠

- **プロジェクトの実績**：寄付総額とGitHub活動は客観的な成長指標
- **コミュニティの拡大**：支援者増加とOSS活動活発化の相関
- **投資性の回避**：価格上昇が「成長」という合理的根拠に基づく設計

### 3.3 トークン移転と手数料

- トークン移転時に**1%をバーン（焼却）**、**0.5%をプラットフォーム収益**とする。
- トークンはXRPL上のIssued Currencyとして発行。
- トークンの外部市場（DEXなど）への上場・流通はプラットフォーム外とし、関与しない。

## 4. 資金調達の流れ（プラットフォーム仲介型）

### 4.1 寄付受付システム

- プラットフォームが**マスターウォレット**を保有し、全ての寄付を一元管理
- 各プロジェクトに**固有のDestination Tag**を割り当て
- 寄付者は「プラットフォームアドレス + プロジェクト固有タグ」宛に送金
- XRPLの**Payment監視システム**により、リアルタイムで寄付を検知・処理

### 4.2 詳細な寄付フロー

1. **寄付実行**：支援者がプラットフォームアドレス（+ Destination Tag）に資金送金
2. **寄付検知**：プラットフォームが自動でPaymentトランザクションを監視・検知
3. **手数料計算**：寄付額から手数料（例：3-5%）を自動計算
4. **トークン発行**：プレマネー評価に基づき、寄付者向けにトークンを新規発行・送付
5. **メンテナー送金**：手数料を差し引いた金額をメンテナーのウォレットに自動送金
6. **記録保存**：全取引履歴をFirestoreに記録（監査・透明性確保）

### 4.3 技術的な実装詳細

- **アトミック処理**：トークン発行とメンテナー送金を同一トランザクション内で実行
- **エラーハンドリング**：送金失敗時の自動リトライ機能
- **手数料透明性**：寄付前に手数料を明示、寄付後に詳細な内訳を提供
- **セキュリティ**：マスターウォレットの秘密鍵は暗号化して安全に管理

### 4.4 寄付に伴う価格・供給量への影響

- 発行に伴いトークン供給が増加し、次回以降の価格算出に影響
- 累積寄付額がプレマネー評価の「客観的評価」要素に反映される可能性

## 5. 貢献者報酬（コントリビュータ向け）

- メンテナーは、貢献者に対してトークンを報酬として配布可能。
- 報酬の形式は以下の2種を選択可能：
  - **即時発行型**：すぐにトークン発行・送付
  - **ベスティング型**：貢献が一定期間・条件を満たした後に権利確定

## 6. トークンのユーティリティ価値（支援証明モデル）

### 6.1 基本設計思想

- トークンは**消費型ではなく保有型**：使い切るのではなく、持ち続けることに価値
- **支援証明**としての機能：どれだけそのOSSを支援したかが一目で分かる
- **ステータス性**：長期保有者ほど「真の支援者」として認識される
- **累積的価値**：保有量に応じて特典レベルが向上

### 6.2 具体的なユーティリティ機能

| 機能              | 内容                                                         | 保有量による変化 |
| ----------------- | ------------------------------------------------------------ | ---------------- |
| 🛠 優先サポート   | トークン保有者のissue対応・リクエストを優先                  | 多いほど高優先度 |
| 💼 商用ライセンス | 商用利用時にトークン所持量によってライセンス料割引（SSPL風） | 段階的割引率向上 |
| 🎖 ステータスNFT  | 支援・貢献に応じたNFTや称号を配布                            | レア度が向上     |
| 🔓 アドオン解放   | 特定機能へのアクセス権（消費ではなく保有量による判定）       | より多機能解放   |
| 🌐 他OSS連携      | 他のOSSトークンとの相互特典・連携が可能                      | 連携範囲拡大     |

### 6.3 商用ライセンス割引の具体例

```text
保有トークン量による割引率：
- 1,000トークン以上：ライセンス料10%割引
- 5,000トークン以上：ライセンス料25%割引
- 10,000トークン以上：ライセンス料50%割引
- 50,000トークン以上：ライセンス料75%割引
```

## 7. メンテナーのメリット設計

- 支援金は全額（手数料除く）受け取れる
- トークンは**OSSの利用・貢献促進を目的としたツール**
- トークン流通によってOSSの価値が可視化され、外部からの評価・連携に繋がる
- コントリビュータの活動に対する報酬を明確に設計可能

## 8. マネタイズ設計（プラットフォーム収益）

| 収益項目           | 内容                                         |
| ------------------ | -------------------------------------------- |
| トークン移転手数料 | 各トランザクションから0.5%                   |
| 支援金からの手数料 | 例：3〜5%（メンテナーに明示）                |
| トークン発行登録料 | 初期費用（任意で設定）                       |
| プレミアム機能課金 | 高度な分析・管理機能の月額課金               |
| トークン運用代行   | トークン管理サポートサービスとして有料化可能 |

## 9. ガバナンス設計

- トークンは基本的に**ガバナンス権限を持たない設計**
- 将来的に導入する場合も、メンテナーが最終実装判断を下せる構造
- 提案権・重み付き投票などの機能は限定的に段階導入可能

## 10. 法的考慮・コンプライアンス

- 本プラットフォームは**トークンをユーティリティとして設計**し、**投資性・証券性を排除**
- 日本の資金決済法およびアメリカの証券法（SEC/Howeyテスト）への対応方針：
  - トークンに利益分配機能・価格保証を持たせない
  - 投資勧誘的な表現を行わない
  - DEXなどの外部市場に関与しない
- AML/KYCが必要な場合に備え、支援金処理部分は監査可能な記録を保持

## 12. アーリーアダプターメリット設計

### 12.1 自然な価格上昇によるメリット

- **早期参入優位性**：プロジェクト初期の低価格時にトークンを取得可能
- **成長の恩恵**：寄付総額とGitHub活動の増加に伴う価格上昇の恩恵を享受
- **長期保有インセンティブ**：時間経過とともにトークンの希少性が向上

### 12.2 投資性回避のための設計

- **価格上昇の根拠**：プロジェクトの客観的成長指標に基づく合理的な価格形成
- **ユーティリティ重視**：価格上昇よりも「支援証明」としての価値を前面に押し出し
- **表現の配慮**：「投資」「リターン」等の表現を避け、「支援」「貢献」を強調

### 12.3 長期支援者への特典

- **保有期間ボーナス**：一定期間以上保有した場合の追加特典
- **初期支援者限定NFT**：プロジェクト開始から一定期間内の支援者のみが受け取れる記念NFT
- **コミュニティ内での地位**：早期支援者としての認知とステータス

## 13. 認証・認可設計

- **認証方式**: Firebase Authを使用したGitHub OAuth認証
- **ユーザー管理**: Firestoreでユーザー情報を管理
- **セキュリティ**: Firestoreセキュリティルールによるデータアクセス制御
- **ウォレット連携**: 認証済みユーザーのみXRPLウォレット連携可能

## 留意事項および今後の課題

### 1. トークン価格形成の透明性と制御

- プレマネー評価ベースの理論価格は**あくまで参考値**として表示し、**市場価格と混同されないよう注意**。
- 将来的に外部市場（DEX等）での流通が発生する可能性があるため、**価格表示と発行ロジックの乖離がないよう設計の整合性**を確保する必要がある。

### 2. 法的リスクの残存可能性

- 投資性・証券性を回避する設計となっているが、**日本および米国では規制が変動的であり、SEC等の判断次第で新たな解釈が加わる可能性**がある。
- 利用規約や説明文で「投資商品ではないこと」を明記し、**言葉遣いやマーケティングにおける表現に留意**する。

### 3. プレマネー評価アルゴリズムの精度と信頼性

- GitHub等の外部指標を元にしたアルゴリズム評価は**運用開始後も継続的に見直し**が必要。
- 評価モデルの**半透明化（ロジックの一部公開）**や、ユーザーからのフィードバック反映機能を将来的に導入することが望ましい。

### 4. トークンの実用性とエコシステム活性化

- トークンの利用価値が実際に存在しないと、**「もらって終わり」の空虚な体験**となり、継続的な支援につながりにくい。
- プロジェクト初期から**明確な特典・ユーティリティのユースケース**を提示し、参加者に価値を実感してもらうことが重要。
